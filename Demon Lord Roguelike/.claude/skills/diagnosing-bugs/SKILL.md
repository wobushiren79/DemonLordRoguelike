# /diagnose —— 调试纪律

排查 bug 一律走固定六步，先理解再动手，禁止凭直觉乱改多个文件。本项目逻辑跨 Handler（单例逻辑）/ Manager（资源）/ Entity（实体）/ Bean（数据）多层，**症状出现的地方往往不是根因所在**，更要靠纪律而非猜测。

## 六步流程

### 1. 复现（Reproduce）
- 先确认**怎样稳定触发**：哪个战斗模式 / 哪张配置表 / 哪步操作 / 什么数据下出现。
- 复现不了就不要改——先和用户对齐复现步骤、用哪个测试场景（见 test-system）、看哪个数值或日志。

### 2. 最小化（Minimize）
- 把复现路径**剥到最小**：去掉无关的 BUFF、敌人、配置项，定位到"改动哪一个变量症状就消失/出现"。
- 优先缩小到单个生物 / 单条配置 / 单次触发，避免在一大堆交互里大海捞针。

### 3. 假设（Hypothesize）
- 基于最小复现，写下**具体的、可证伪的**根因假设（如"BUFF 周期剩余次数在 X 处少减了一次"）。
- 同时列 2~3 个候选，按可能性排序；不要只押一个。

### 4. 插桩（Instrument）
- 用日志 / 断点 / 打印关键中间值来**验证或推翻**假设，而不是直接改代码看效果。
- 关注数据流转节点：Bean 的取值（如 `CreatureBean.GetAttribute`）、事件派发（EventHandler / BuffEventDispatcher）、管线处理（ModifierPipeline）等。
- 插桩用的临时日志，修复后清理（参照 CLAUDE.md「临时脚本清理规则」的精神）。

### 5. 修复（Fix）
- 只有当插桩**确证根因**后才动手改。
- 改**根因**而非掩盖症状；改最小范围，避免顺手重构引入新问题。
- 改完同步更新对应 XML 注释（[feedback_comment_sync](../../memory/feedback_comment_sync.md)）；若命中 `watched_files` 同步 Agent/Skill 文档。

### 6. 测试（Test）
- 回到第 1 步的复现路径，确认症状消失。
- 再验证**没有引入回归**：相关模块（BUFF/攻击/AI/奖励等常互相牵连）的既有行为是否仍正常。
- 必要时用 `/verify` 或对应测试场景实跑确认。

## 反模式（禁止）

- **跳过复现直接改**：没稳定复现就改，等于盲修。
- **猜测式扫射**：一次改多个文件"看看哪个管用"——根因不明的改动只会制造新 bug。
- **改症状不改根因**：在症状点加 if 绕过，根因仍在，换个场景复发。
- **不验回归**：只确认原 bug 没了，不检查有没有弄坏别的。

## 输出

排查结束时向用户报告：**根因定位**（在哪个文件哪一层）、**为什么会错**、**改了什么**、**怎么验证的**。不确定的部分如实说明，不要把"可能修好了"说成"已修复"。
