# /grill —— 动手前的需求盘问循环

把"看一眼就猜着改"换成"先问清所有决策分支再动手"。核心是一个**反复追问的访谈循环**：在所有会影响实现的未定项被解决之前，不写代码、不改配置。

## 何时启用

**自动启用（model-invoked）**——当需求命中以下任一高歧义/高风险信号且存在未定项时，先盘问再动手：

- 改**配置表结构**（新增/删除字段、改字段含义、改 id 规则）。
- 调**战斗 / BUFF / 数值**（堆叠策略、触发条件、成功率公式、掉落概率等一处改动会连锁影响别处）。
- **改变架构或数据流向**（配置来源 Excel⇄代码字段迁移、Handler/Manager 职责调整）——此项与 [feedback_ask_before_architecture_change](../../memory/feedback_ask_before_architecture_change.md) 叠加，必须先问。
- **跨多个模块**的改动，或需求里出现"等等/类似的/顺便"这类边界不清的措辞。

**显式启用**——用户输入 `/grill`，对当前讨论的计划/设计做盘问。

> 反向约束：简单、单点、低风险的改动（改一句文案、补一个注释、明确的单文件小修）**不要**触发盘问，直接做。盘问是为高歧义需求服务的，不是每次都问。

## 盘问循环

重复以下步骤，直到**没有未解决的决策分支**为止：

1. **列未定项**：把需求里所有"还没定"的点显式列出来——边界、取值范围、异常分支、与现有数据/存档的兼容、影响到的其他模块。
2. **逐条提问**：一次问一组相关问题，优先用 `AskUserQuestion`（选项要给推荐项并标"(推荐)"）。问题与选项**一律中文**。每个问题都要能改变接下来怎么做——不问对实现没影响的问题。
3. **收敛**：用户回答后，更新未定项清单；回答若引出新分支，继续追问。
4. **复述确认**：所有分支解决后，用一段话把最终方案复述给用户（要改哪些文件/表、取什么值、影响哪些模块），等用户确认。
5. **才动手**：确认后再进入实现；实现中若又冒出未定项，回到第 1 步。

## 提问要覆盖的维度（清单）

- **数据真实源**：要改的是 Excel 源表还是 JSON 派生数据？（配置表唯一真实源是 Excel，见 [reference_language_excel_source](../../memory/reference_language_excel_source.md) 与 CLAUDE.md「Excel 是唯一真实源」）
- **取值与边界**：范围、默认值、上下限、为空/为 0 时的行为。
- **兼容性**：老存档、已有配置行、已生成的 Bean 是否受影响？要不要迁移？
- **连锁影响**：还有哪些模块/配置表/枚举引用了它？（BUFF、攻击模式、奖励、AI 等常互相牵连）
- **范围边界**：这次改到哪为止？哪些明确**不**改？
- **验收**：怎么算改对了？有没有可复现的验证方式（跑哪个测试场景 / 看哪个数值）？

## 失败模式（避免）

- **过早完工**：未定项还没问完就开写——这是本 skill 要消灭的首要问题。
- **问废话**：问对实现没有影响的问题，徒增往返。
- **一次塞太多**：一口气抛十个问题。一次问一组相关的，让用户好回答。
- **不复述就动手**：跳过第 4 步，导致理解偏差到实现后才暴露。

## 与项目其他规则的衔接

- 涉及改架构/数据流向 → 叠加 CLAUDE.md「Bean 修改规则」「Excel 唯一真实源」，必须先问。
- 盘问产出的关键约定若具长期价值，按 CLAUDE.md「记忆系统」写入 `.claude/memory/` 并更新 `MEMORY.md`。
