# Code Review Skill

## 指令
`/code-review`

## 描述
代码审查技能，用于审查项目代码质量，发现潜在问题并提供改进建议。

## 使用方式

### 1. 审查未提交的代码（默认）
```
/code-review
```
当不提供任何参数时，自动审查当前未提交的代码变更（通过 git diff 获取）。

### 2. 审查指定模块
```
/code-review <模块名>
```
审查指定模块的代码。例如：
- `/code-review AI` - 审查 AI 模块
- `/code-review Game` - 审查 Game 模块
- `/code-review MVC` - 审查 MVC 模块

## 可用模块列表
- `AI` - AI相关逻辑
- `Bean` - 数据对象/实体类
- `Common` - 公共代码
- `Component` - 组件代码
- `Enums` - 枚举定义
- `Game` - 游戏核心逻辑
- `MVC` - MVC框架代码
- `Struct` - 结构体定义
- `Utils` - 工具类

## 审查流程

### 当用户调用 `/code-review` 时：

1. **判断审查模式**
   - 如果没有参数：执行未提交代码审查
   - 如果有参数：执行指定模块审查

2. **未提交代码审查流程**：
   ```
   a. 执行 git status 查看变更文件
   b. 执行 git diff 获取具体变更内容
   c. 对变更进行代码审查
   ```

3. **指定模块审查流程**：
   ```
   a. 确认模块名称是否有效
   b. 扫描 Assets/Scripts/<模块名>/ 目录下的所有 .cs 文件
   c. 对模块代码进行全面审查
   ```

## 审查标准

### 代码质量检查项

1. **命名规范**
   - 类名使用 PascalCase
   - 方法名使用 PascalCase
   - 私有字段使用 _camelCase 或 m_camelCase
   - 常量使用 UPPER_CASE 或 PascalCase
   - 避免使用缩写，除非是通用的（如 UI, ID）

2. **代码结构**
   - 单一职责原则
   - 方法长度适中（建议不超过50行）
   - 类长度适中（建议不超过500行）
   - 避免深层嵌套（建议不超过3层）

3. **Unity 特定规范**
   - 避免 Update() 中的 Find/GetComponent 调用
   - 使用 SerializeField 而非 public 字段
   - 正确使用 MonoBehaviour 生命周期
   - 避免在循环中分配内存

4. **性能考虑**
   - 避免不必要的字符串拼接
   - 使用 StringBuilder 处理大量字符串操作
   - 注意装箱拆箱操作
   - 合理使用对象池

5. **安全性**
   - 空值检查
   - 数组边界检查
   - 异常处理

6. **可维护性**
   - 适当的注释
   - 代码可读性
   - 避免魔法数字
   - 合理的代码组织

## 示例

### 示例1：审查未提交代码
```
用户: /code-review

助手执行:
1. git status
2. git diff
3. 分析变更并生成审查报告
```

### 示例2：审查指定模块
```
用户: /code-review Game

助手执行:
1. 扫描 Assets/Scripts/Game/ 目录
2. 读取所有 .cs 文件
3. 进行全面代码审查
4. 生成审查报告
```

## 注意事项

1. 审查应该是建设性的，既要指出问题也要认可优点
2. 问题按严重程度分类，便于开发者优先处理
3. 每个问题都要提供具体的修复建议
4. 审查结果以 Markdown 格式输出，便于阅读
